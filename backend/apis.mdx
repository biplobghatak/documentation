This page catalogues the **internal** route handlers under `app/api/*` â€” the
endpoints Spyro's own browser app calls. They authenticate with the Supabase
session cookie (not an API key), and the [middleware](/backend/middleware) leaves
`/api` open so each handler enforces its own auth.

<Note>
The externally-consumable surface is documented separately in
[Public API](/backend/public-api). This page covers everything else.
</Note>

## Auth conventions

Internal handlers use one of a few patterns:

| Pattern | Meaning | Typical failure |
| --- | --- | --- |
| `getCurrentUser()` | Requires a signed-in user | `401` or `redirect("/login")` |
| `requireAccess(orgSlug)` | Requires org membership | `403` / `404` |
| `resolveWorkspaceForUser(workspaceId)` | Requires workspace/org access | `404` |
| **Public** | No auth â€” used by marketing/free tools | per-IP rate limit |
| Webhook signature | Verifies a provider HMAC | `400` invalid signature |
| Admin session | The `spyro_admin` cookie + TOTP | `404` (hidden) |

A recurring rule: **long, durable work goes to [Inngest](/backend/background-jobs)**;
short background work on public tools uses Next's `after()` with client polling.
The "Inngest?" column below flags which is which.

## Onboarding & site setup

These power the signup wizard. All require a signed-in user (`getCurrentUser()`)
unless noted.

| Method Â· Path | Purpose | Auth | Inngest? |
| --- | --- | --- | --- |
| `POST /api/onboarding/quick-scan` | Tier-1 single homepage fetch (<3s): brand name, colors, logo, sitemap, language | user | no |
| `POST /api/onboarding/derive` | Tier-2 parallel LLM audience derivation + competitor discovery (fault-isolated) | user | no |
| `POST /api/onboarding/competitors` | Re-fetch competitors with locale + business context | user | no |
| `POST /api/onboarding/seed-prompts` | Draft AI-visibility tracked prompts for wizard review | user | no |
| `GET /api/site-brand` | Best-effort brand name + logo; always returns a fallback | **public** | no |
| `GET /api/site-index/{siteId}` | Poll onboarding index progress + fan-out track statuses | workspace access | no |

`POST /api/onboarding/derive` returns audience tags, a business description, and
competitors, with each half fault-isolated so a slow DataForSEO call can't block
the audience result. `GET /api/site-index/{siteId}` reports `seo`, `geo`, `serp`,
and `intel` track statuses for the post-onboarding progress UI; it is gated on
**org** access (not `user_id`) so team members and seeded workspaces can poll.

<Warning>
`GET /api/site-brand` is **public** and fetches an arbitrary user-supplied
domain server-side. It is SSRF-guarded and falls back to a domain-derived name +
favicon on any failure (returns `200` with `ok: false`). See [Security](/backend/security).
</Warning>

## Free public tools

The marketing site's lead-gen tools. All are **public**, per-IP rate-limited, and

| `POST` Â· `GET /api/free-audit` | Full free SEO/GEO site audit | 8 / 10 min |

Each cached-report tool has a sibling **PDF route** (`/api/{tool}/pdf?website=â€¦`)

<Info>

`POST /api/free-audit/subscribe` validates the email, stores the lead

## Integrations: GA4 & GSC OAuth

Google Analytics 4 and Google Search Console connect via OAuth. Every route
resolves the workspace with `resolveWorkspaceForUser(workspaceId)`.

| Method Â· Path | Purpose | Inngest? |
| --- | --- | --- |
| `GET /api/ga4/connect` Â· `GET /api/gsc/connect` | Start OAuth â€” redirect to Google with a signed, CSRF state cookie (600s TTL) | no |
| `GET /api/ga4/callback` Â· `GET /api/gsc/callback` | Validate state, exchange code for tokens, store integration, redirect to property picker | no |
| `POST /api/ga4/select-property` Â· `POST /api/gsc/select-property` | Verify the chosen property against Google, store it | **yes** |
| `POST /api/ga4/disconnect` Â· `POST /api/gsc/disconnect` | Best-effort revoke at Google, then delete the integration row | no |

The `connect` routes return `503` when OAuth isn't configured
(`featureFlags.ga4OauthConfigured` / `gscOauthConfigured`). The `select-property`
routes are the only ones here that **emit Inngest events** â€”
`"workspace/ga4.connected"` / `"workspace/gsc.connected"` â€” to trigger the
on-connect data import. GSC's `select-property` also returns a `domainMismatch`
flag. Token exchange failures surface as `502`; a property the account can't
access is `403`.

## Billing & subscription

Spyro bills through **Dodo Payments** (not Polar â€” the README is stale). See
[Billing](/backend/billing).

| Method Â· Path | Purpose | Auth | Errors |
| --- | --- | --- | --- |
| `GET /api/checkout` | Create a Dodo checkout session and redirect to it | user | `400` bad plan/interval/sites, `404`, `500` |
| `GET /api/portal` | Open the Dodo customer portal | user | `400` no billing account, `500` |
| `GET /api/subscription/status` | `{ active }` â€” polled after checkout | user | `401` |
| `POST /api/subscription/update` | Change plan/interval/site count in place | `requireAccess`, **owner** | `403` non-owner, `409` no sub yet, `502` Dodo error |
| `POST /api/webhooks/dodo` | Dodo webhook â€” reconcile subscription state, send billing emails | **signature** | `400` invalid signature |

`GET /api/checkout` takes `plan` (`pro`|`agency`), optional `interval`
(`month`|`year`), and optional `sites`. `POST /api/subscription/update` is
**owner-only** and writes the org optimistically; the webhook reconciles from
Dodo metadata.

### The Dodo webhook

`POST /api/webhooks/dodo` is authenticated by **signature verification**, not a
session â€” it reads the raw body and calls `verifyDodoWebhook(req, raw)`, returning
`400` on a bad signature. It then routes on event type:

| Event(s) | Effect |
| --- | --- |
| `subscription.active` / `renewed` / `updated` / `plan_changed`, `payment.succeeded` | Activate the plan; re-sync plan/interval/sites from `metadata`; welcome email on first activation only |
| `subscription.cancelled` / `failed` / `expired` | Deactivate the plan; a cancellation email (with access-end date) is sent for `subscription.cancelled` only |
| `payment.failed` | Payment-failed email (card last-4, amount) |

The org is keyed by `metadata.org_id`; a missing `org_id` is logged and returns
`200` (not an error). Billing emails are best-effort and never fail the webhook.

## Admin

The superadmin endpoints. Gated by IP allowlist + the `spyro_admin` session;
failures return `404` to hide the surface. See
[Authorization](/backend/authorization#superadmin--impersonation).

| Method Â· Path | Purpose | Notable |
| --- | --- | --- |
| `POST /api/admin/login` | Username + scrypt password + TOTP login | `429` per-IP lockout (5 fails / 15 min); sets `spyro_admin` cookie (1h, `sameSite=strict`) |
| `POST /api/admin/logout` | Clear the admin cookie | de-escalating, no gate |
| `POST /api/admin/impersonate` | Start impersonating `targetUserId` (mint magic link, `verifyOtp`) | admin session; `404` if not admin; returns `redirectTo` |
| `POST /api/admin/end-impersonation` | End impersonation, sign out, mark audit row | de-escalating; clears `spyro_impersonation` |

All three credential factors in `/api/admin/login` are evaluated without
early-return to avoid timing leaks, and a single generic `"Invalid credentials."`
is returned regardless of which factor failed.

## Error-code conventions

| Status | Meaning across internal routes |
| --- | --- |
| `400` | Bad input / validation / SSRF rejection |
| `401` | Not signed in |
| `402` | Quota/credit limit (AI credits, plan limits) |
| `403` | Signed in but lacks the required role/membership |
| `404` | Resource not found, or hidden (admin surface) |
| `429` | Rate limited (per-IP free tools, AI credits, admin lockout) |
| `502` | Upstream provider failure (Google, Dodo) |
| `503` | Feature unconfigured or server busy (concurrency cap) |
