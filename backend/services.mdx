Almost all of Spyro's domain logic lives under `lib/*` as small, mostly-pure,
vendor-agnostic modules. Route handlers, server actions, and Inngest functions are
thin - they call into these services. This page is the **directory**: what each
module is responsible for, and where to read more.

<Note>
  Several services share a selector-behind-an-interface shape (a `getXProvider()`-style
  factory), but their **keyless fallback differs - don't assume a mock**. Only
  `lib/site-intel` degrades to a deterministic mock when unkeyed. `lib/llm`, `lib/serp`,
  and `lib/images` **throw** when no provider key is configured (no mock fallback);
  `lib/publish` is a static registry of CMS adapters; and `lib/vector` always returns the
  single pgvector store. Check each module's own page for its exact behavior.
</Note>

## AI, search, and content engines

These are the heavy subsystems; each has its own deep-dive page.

| Module | Responsibility | Deep dive |
|---|---|---|
| `lib/llm` | The internal LLM Spyro **uses** (parse, classify, write). OpenRouter-first provider abstraction with tiers + `generateJson`. | [AI Engine](/backend/ai) |
| `lib/engines` | The external "citation-target" AIs Spyro **queries** (ChatGPT/Claude/Gemini/Perplexity/AI Overview) to measure citations. | [AI Engine](/backend/ai) |
| `lib/agent` | The Spyro chat agent - model selection, system prompt, tools, RAG grounding, turn routing, conversation persistence. | [AI Engine](/backend/ai) |
| `lib/citations` | Runs a prompt against an engine N times, parses cited/competitor domains, aggregates to a frequency; seeds buyer-question prompts. | [AI Engine](/backend/ai) |
| `lib/embed` | OpenAI embeddings (`text-embedding-3-small`), the HTML chunker, and knowledge-document builders for RAG. | [AI Engine](/backend/ai) |
| `lib/vector` | pgvector store over `site_chunks` - upsert/replace by namespace, cosine-similarity search. | [AI Engine](/backend/ai) · [Database](/backend/database) |
| `lib/rerank` | Reranks RAG candidates via an OpenRouter-hosted Cohere reranker, falling back to vector order. | [AI Engine](/backend/ai) |
| `lib/seo` | Keyword difficulty/intent/clustering/opportunity/rankability, issue detection, health scoring, content plan. | [SEO Engine](/backend/seo-engine) |
| `lib/serp` | `getSerpProvider()` - DataForSEO SERP results + features (no mock fallback). | [SEO Engine](/backend/seo-engine) |
| `lib/dataforseo` | Shared DataForSEO HTTP client: auth, retries, DataForSEO-aware errors, daily-budget circuit breaker. | [SEO Engine](/backend/seo-engine) |
| `lib/keyword-data` | Keyword metrics (volume/cpc/difficulty) via DataForSEO Labs with a heuristic mock fallback + cache/freshness. | [SEO Engine](/backend/seo-engine) |
| `lib/geo` | The 16-point GEO/AEO readiness check + weighted GEO score. | [GEO Engine](/backend/geo-engine) |
| `lib/crawler` | Same-origin Cheerio crawler: discovery, robots/sitemap, fetch + optional JS render, parsing, limits. | [Crawler](/backend/crawler) |
| `lib/site-intel` | Off-domain competitive intelligence (backlinks, DR, ranked keywords, SERP competitors) via DataForSEO or mock. | [Crawler](/backend/crawler) |
| `lib/discovery` | Maps persisted blog-idea rows into content-calendar candidates. | [Content Engine](/backend/content-engine) |
| `lib/url` | `normalizeDomainInput` - shape-level domain validation (defers IP/DNS checks to the SSRF guard). | [Crawler](/backend/crawler) · [Security](/backend/security) |
| `lib/audit` | Site/page audit helpers: per-page report rows, template billing, issue labels, severity summaries, audit diffs. | [Audit](/backend/audit) |
| `lib/writer` | Article-writing engine: `enqueueArticleWrite`, multi-phase `writeArticleAdvanced`, quality gates. | [Content Engine](/backend/content-engine) |
| `lib/research` | Live web research (`researchTopicLive` - Kimi `:online` + Jina Reader) producing a `ResearchPack`. | [Content Engine](/backend/content-engine) |
| `lib/publish` | Publish-destination registry (WordPress.org/.com, Notion) + `markdownToHtml`. | [Content Engine](/backend/content-engine) · [Integrations](/backend/integrations) |

## Billing, usage, and plans

| Module | Responsibility | Deep dive |
|---|---|---|
| `lib/plans.ts` | Plan/pricing source of truth - `PLANS` catalog, pricing math, `planLimits`, `blogQuota`, `availableEngines`, `withinLimit`. | [Billing](/backend/billing) |
| `lib/billing` | The billing gate - `hasBillingAccess(org)` (true only on an active subscription, never the legacy local trial). | [Billing](/backend/billing) |
| `lib/dodo` | Dodo Payments client - checkout, subscription change, customer portal, `verifyDodoWebhook`. | [Billing](/backend/billing) |
| `lib/usage.ts` | Metered-usage accounting - `getUsage`/`addUsage`/`checkUsage` over named metrics (AI credits, citation calls, tracked keywords, audit pages). | [Billing](/backend/billing) |

<Warning>
  Billing is **Dodo Payments, not Polar.** Env vars are `DODO_*`, the webhook route
  is `app/api/webhooks/dodo`, and the migration is `0022_dodo_payments`. README
  references to Polar are stale - the code wins. See [Billing](/backend/billing).
</Warning>
